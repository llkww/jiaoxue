请严格按照下面这套规范绘制 ER 图，不要使用你自己的随意记法，必须遵守以下规则：

一、总目标
1. 先从项目所有代码、业务描述中提取：
   - 实体集
   - 每个实体集的属性
   - 主码
   - 联系集
   - 联系上的属性
   - 角色名（如果同一实体在同一联系中多次参与）
   - 基数约束
   - 全参与 / 部分参与
   - 弱实体及其依赖的强实体
2. 然后输出一张结构清晰、符号规范、可直接用于数据库课程作业/课程设计的 ER 图。
3. 如果业务描述信息不足，不要擅自乱补；可以基于最合理假设补全，但必须明确标注“以下约束为推断”。

二、ER 图绘制规范（必须严格执行）
1. 实体集：
   - 每个实体集使用“上下两个矩形”表示。
   - 上方矩形写实体名。
   - 下方矩形写属性列表。
   - 主码属性必须加下划线。

2. 联系集：
   - 每个联系集用菱形表示。
   - 联系名称写在菱形内部。
   - 如果联系本身有属性，则将这些属性从菱形引出并单独标注。

3. 角色名：
   - 如果同一个实体集在同一个联系中以不同身份参与，必须在对应连线上标注角色名。
   - 例如先修课关系中，同一个 Course 实体可以分别以 course_id 和 prereq_id 两种角色参与同一个联系。

4. 基数约束：
   - 必须对每个联系两端标明基数约束。
   - 按以下规则表达：
     - “1” 用箭头表示
     - “多” 用普通无向线段表示
   - 要正确表达以下四类关系：
     - 一对一
     - 一对多
     - 多对一
     - 多对多

5. 参与约束：
   - 部分参与：单线
   - 全参与：双线
   - 若更适合，也可以在连线旁显式标注 l..h 形式的参与范围，如：
     - 0..*
     - 1..1
     - 1..*
   - 若使用 l..h，则要保证最小参与次数和最大参与次数语义正确。

6. 弱实体集：
   - 弱实体集必须使用双线矩形表示。
   - 弱实体的区别属性集用虚线标记。
   - 弱实体不能单独存在，必须通过识别联系依赖某个强实体。
   - 绘图时要体现这种依赖关系，不可把弱实体画成普通强实体。

三、输出要求
1. 先给出“实体、属性、主码、联系、基数、参与约束、弱实体识别”的整理结果。
2. 再生成 ER 图。
3. 图中所有文字使用数据库中的英文命名，保持统一。
4. 图形布局要整齐、对齐、避免线条交叉，保证可读性强。
5. 如果一个联系有联系属性，必须清楚地挂接在菱形上，而不是误放到实体属性里。
6. 如果存在递归联系或自反联系，必须明确标注角色名，避免歧义。
7. 不要漏掉主码下划线。
8. 不要把基数约束和参与约束混为一谈，两者都要表达。

四、如果需要附加输出
在画完 ER 图后，再补充一段“ER 图转关系模式”的结果，转换规则如下：
1. 强实体集可直接转为关系模式。
2. 弱实体集转关系模式时，要包含自身属性和所依赖强实体的主码。
3. 联系集转关系模式时，要包含联系自身属性以及参与联系的实体主码。

五、实现形式
优先生成 HTML + SVG 的静态可视化代码，要求：
- 结构清晰
- 文字不重叠
- 连线准确
- 矩形、双线矩形、菱形、双线、虚线、箭头都能精确绘制
- 输出完整可运行代码
