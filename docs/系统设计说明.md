# 教学管理系统系统设计说明

## 1. 文档目的

本文档基于当前代码实现，对教学管理系统的架构、模块、数据库、关键业务规则、前后端交互方式和初始化方案进行系统说明。文档目标不是描述“理想中的系统”，而是准确反映当前仓库中的真实实现，便于：

- 课程设计答辩
- 后续维护与迭代
- 新成员快速理解代码结构
- 校验文档与代码是否一致

## 2. 系统概述

### 2.1 建设目标

系统面向高校教学管理场景，围绕学生、教师、管理员三类角色，构建统一的数据底座和统一的业务规则，实现以下核心目标：

- 统一账号体系与基于角色的访问控制
- 课程、开课、选课、成绩、评价、公告、培养方案等核心教务链路贯通
- 前端页面状态、后端业务校验、数据库约束三层一致
- 支持完整的演示数据初始化，便于课程设计展示

### 2.2 角色范围

系统当前支持三类角色：

- 学生
- 教师
- 管理员

三类角色共享同一个 `users` 账号表，但通过扩展表区分个人档案：

- 学生对应 `students`
- 教师对应 `teachers`
- 管理员对应 `admins`

### 2.3 核心业务对象

系统围绕以下核心对象展开：

- 学期
- 院系
- 专业
- 班级
- 用户与身份档案
- 课程
- 培养方案
- 培养方案模块
- 培养方案课程映射
- 开课记录
- 选课记录
- 成绩记录
- 教学评价
- 学业预警
- 公告

## 3. 技术架构

### 3.1 整体架构

系统采用传统的服务端渲染架构，整体分为四层：

1. 表现层
2. 路由控制层
3. 服务与工具层
4. 数据持久层

对应到代码目录：

```text
views / public        -> 表现层
routes                -> 路由控制层
services / utils      -> 服务与工具层
config / sql / mysql  -> 数据持久层
```

### 3.2 技术选型

#### 后端

- Node.js
- Express 4
- express-async-errors
- mysql2/promise
- express-session
- express-mysql-session
- bcryptjs
- method-override

#### 前端

- EJS
- Bootstrap 5
- 原生 JavaScript
- Chart.js
- Iconify
- 自定义 CSS 设计系统

#### 数据库

- MySQL 8.0
- `utf8mb4`

### 3.3 为什么采用该方案

当前架构适合课程设计与教务场景原型，主要原因如下：

- 服务端渲染能直接把权限、页面布局和数据库结果统一输出
- Express 路由清晰，便于按角色拆分功能
- MySQL 适合实现实体关系、唯一约束、外键和复杂多表查询
- 无前端构建链路，便于在 Windows 本地和课程答辩环境中快速运行
- 使用会话而不是 JWT，适合服务端模板项目

## 5. 请求处理流程

### 5.1 应用启动流程

`app.js` 中的启动过程如下：

1. 载入 `express-async-errors`
2. 创建 Express 实例
3. 配置视图目录、布局模板
4. 注册：
   - URL 编码表单解析
   - JSON 解析
   - `method-override`
   - 静态资源目录
   - Session 中间件
   - Flash 中间件
   - locals 中间件
5. 挂载业务路由
6. 挂载 404 / 500 处理器
7. 启动监听并执行数据库 `ping`

### 5.2 页面请求流程

一个典型页面请求的处理链路如下：

1. 浏览器请求页面
2. Express 进入中间件链
3. Session 从 MySQL 中恢复登录态
4. `flashMiddleware` 读取并消费一次性提示
5. `localsMiddleware` 写入：
   - 当前用户
   - 当前路径
   - 当前学期
   - 动态导航
   - 格式化函数
6. 路由层执行数据库查询和业务判断
7. EJS 渲染页面
8. 页面加载公共 JS，绑定确认弹层、弹窗表单、AJAX 保存、筛选增强等交互

### 5.3 写操作流程

对于新增、编辑、删除等写操作，系统通常采用以下策略：

1. 路由层先做参数规范化
2. 路由层做业务规则校验
3. 需要跨表更新时使用事务
4. 成功后写入 `req.flash('success', ...)`
5. 失败后写入 `req.flash('danger', ...)`
6. 重定向回来源页或安全回跳地址

## 6. 权限与会话设计

### 6.1 会话存储

系统采用：

- `express-session`
- `express-mysql-session`

会话特征如下：

- Cookie 名：`tm.sid`
- 存储位置：MySQL Session 表
- 有效期：7 天
- `httpOnly: true`
- `sameSite: 'lax'`
- 生产环境支持 `secure`

### 6.2 认证中间件

#### `requireAuth`

未登录时：

- 写入 warning Flash
- 跳转到 `/auth/login`

#### `requireRoles(...roles)`

未登录时：

- 同样先跳转登录

角色不匹配时：

- 返回 403 页面

#### `redirectIfAuthenticated`

已登录用户访问登录页时：

- 直接跳转到 `/dashboard`

### 6.3 会话用户对象

登录成功后，`services/userService.js` 会查询并组装统一的会话用户对象，包括：

- `id`
- `username`
- `role`
- `fullName`
- `email`
- `phone`
- `avatarColor`
- `status`
- `profileId`
- `identityCode`
- `meta`

其中：

- 学生的 `identityCode` 是学号
- 教师的 `identityCode` 是工号
- 管理员的 `identityCode` 是管理员编号

## 8. 前端设计与交互方案

### 8.1 布局设计

系统采用统一后台布局：

- 左侧固定导航
- 顶部状态栏
- 中央内容区
- Flash 提示区

布局模板文件为：

- `views/layout.ejs`
- `views/auth-layout.ejs`

### 8.2 页面渲染方式

系统以 EJS 服务端渲染为主，前端 JS 负责增强交互，而不是完全接管页面状态。

这种设计有几个优点：

- 页面初次渲染简单直接
- 权限菜单和状态标签容易统一
- 表单回跳和 Flash 提示更自然
- 适合课程设计演示和后端主导型系统

### 8.3 公共前端交互

`public/js/app.js` 负责以下前端增强：

- 自定义确认弹层
- 提交按钮加载态
- 成功状态短暂回放
- 页面滚动位置恢复
- Flash 自动关闭
- 受保护删除二次确认
- 自定义筛选下拉
- 院系-专业联动
- 管理端弹窗式表单编辑
- 培养方案模块详情弹窗
- 学生评价弹窗
- 编号预览
- 成绩 AJAX 保存
- 图表初始化

### 8.4 管理端弹窗编辑机制

管理端新增和编辑功能大量采用弹窗编辑器，处理流程如下：

1. 点击带 `data-admin-modal` 的链接
2. 前端通过 `fetch` 请求表单页面
3. 从返回 HTML 中抽取 `.editor-shell`
4. 注入通用弹窗容器
5. 用户在弹窗内提交表单
6. 前端以 `XMLHttpRequest` 方式提交
7. 服务端如果需要跳转，不直接发真实重定向，而是返回：
   - `X-Redirect-To`
   - `X-Redirect-Status`
8. 前端读取头信息后做一次真实页面跳转

这样做的目的，是解决“Flash 提示被隐藏请求提前消费，用户需要刷新两次才看到错误”的问题。

### 8.5 培养方案前端展示

学生端培养方案不是简单表格，而是组合了：

- 总体完成度
- 已修 / 在修 / 未通过统计
- 思维导图式学期节点布局
- 模块详情弹窗
- 未通过课程高亮

当前样式通过 `public/css/app.css` 中的对称主干布局实现，根节点会连接到左右学期节点，各模块再连接到对应学期。

## 10. 工具层设计

### 10.1 `utils/system.js`

定义统一枚举：

- 角色
- 用户状态
- 学期状态
- 课程类型
- 开课状态
- 选课状态
- 成绩状态
- 公告优先级
- 公告类别

并提供统一的标签与徽章样式映射。

### 10.2 `utils/identity.js`

负责编号生成规则：

#### 学号

格式：

```text
院系编号2位 + 入学年份后2位 + 班级编号2位 + 班内序号2位
```

#### 工号

格式：

```text
T + 院系编号2位 + 当前年份后2位 + 2位流水号
```

#### 课程号

格式：

```text
专业代码 + R/E + 3位流水号
```

其中：

- `R` 表示必修
- `E` 表示选修

#### 开课编号

格式：

```text
T + 学期ID两位 + '-' + 课程号 + '-' + 2位流水号
```

### 10.3 `utils/score.js`

负责总评、等级和绩点计算：

#### 总评

```text
总评 = 平时分 * 平时占比 + 期末分 * 期末占比
```

#### 等级

- `>= 90`：A
- `>= 80`：B
- `>= 70`：C
- `>= 60`：D
- `< 60`：F

#### 绩点

- `>= 90`：4.0
- `< 60`：0
- `60~89`：每增加 1 分增加 `0.1`

### 10.4 `utils/pagination.js`

统一分页：

- 页面号最小值为 1
- `pageSize` 最大值限制为 30
- 分页条窗口大小为 6

### 10.5 `utils/schedule.js`

负责把上课时间段转换成二维课表网格，供：

- 学生我的课表
- 教师课表
- 单课程表详情

使用。

16. `course_sections`
17. `enrollments`
18. `grades`
19. `teaching_evaluations`
20. `academic_warnings`

### 11.2 关系设计

主要关系如下：

- 一个院系对应多个专业
- 一个专业对应多个班级
- 一个用户只能对应一个学生 / 教师 / 管理员档案
- 一个专业最多一个培养方案
- 一个培养方案下有多个模块
- 一个模块下有多个课程映射
- 一门课程可在多个学期开设多个开课记录
- 一个开课记录可对应多个选课记录
- 一个选课记录唯一对应一个成绩记录
- 一个选课记录最多对应一条教学评价

### 11.3 关键约束

系统设计明确要求避免：

- 删除异常
- 增加异常
- 修改异常
- 级联误删

因此当前 schema 采取以下策略：

- 大部分外键使用 `ON DELETE RESTRICT`
- 少数展示性弱引用使用 `SET NULL`
- 不使用“自动级联删除业务数据”的设计
- 删除动作先由业务层做依赖检查

### 11.4 索引与唯一约束

关键索引示例：

- `terms (is_current, start_date)`
- `majors (department_id, name)`
- `courses (department_id, major_id)`
- `courses (major_id, department_id)`
- `course_sections (term_id, selection_status)`
- `course_sections (course_id, term_id)`
- `enrollments (student_id, status)`
- `grades (status, total_score)`
- `teaching_evaluations (teacher_id, created_at)`

关键唯一约束示例：

- 学年 + 学期标签唯一
- 学号唯一
- 工号唯一
- 课程号唯一
- 同一学期教师同时间唯一
- 同一学期教室同时间唯一
- 同一学生同一开课唯一
- 同一选课唯一成绩记录
- 同一选课唯一评价记录
- 同一培养方案中课程唯一

## 12. 核心业务流程设计

### 12.1 登录流程

1. 用户访问 `/auth/login`
2. 提交账号、密码、角色
3. 后端校验：
   - 账号是否存在
   - 角色是否匹配
   - 账号是否启用
   - 密码哈希是否匹配
4. 更新最近登录时间
5. 查询并组装会话用户对象
6. 写入 session
7. Flash 提示欢迎信息
8. 跳转 `/dashboard`

### 12.2 学生选课流程

1. 学生进入在线选课页
2. 系统读取当前学期
3. 只显示当前学期、开放选课状态的开课
4. 选课时依次检查：
   - 当前学期是否存在
   - 课程是否属于当前学期
   - 开课是否开放
   - 容量是否已满
   - 是否与已选课程时间冲突
5. 写入或恢复 `enrollments`
6. 新建或重置对应 `grades`
7. 返回课程页并提示结果

### 12.3 学生退课流程

1. 检查选课记录是否属于当前学生
2. 检查是否为“已选”状态
3. 检查是否为当前学期
4. 检查是否已经形成已发布及格成绩
5. 将 `enrollments.status` 改为 `已退课`
6. 清空该条成绩记录内容并恢复为 `待录入`

### 12.4 教师成绩录入流程

1. 教师进入某门课成绩册
2. 输入平时分和期末分
3. 后端按该开课当前占比计算：
   - 总评
   - 绩点
   - 等级
4. 保存到 `grades`
5. 状态保持为 `待录入`
6. 前端 AJAX 回填行内总评与绩点

### 12.5 教师成绩发布流程

1. 教师点击发布
2. 系统查询当前开课下：
   - 学生数量
   - 已有总评数量
3. 把所有有总评的成绩状态更新为 `已发布`
4. 返回详情页并提示发布成功

### 12.6 管理员开课维护流程

1. 管理员先维护课程、教师、学期、教室、时间段
2. 创建或编辑开课时输入：
   - 课程
   - 教师
   - 学期
   - 教室
   - 时间段
   - 容量
   - 占比
3. 系统检查：
   - 教师占用冲突
   - 教室占用冲突
   - 成绩占比之和是否为 100
4. 写入 `course_sections`
5. 如果编辑了占比，则联动重算成绩

### 12.7 培养方案维护流程

1. 管理员创建培养方案
2. 在方案下维护模块
3. 在模块下关联课程
4. 系统同步：
   - 模块要求学分
   - 培养方案总学分
   - 相关学生毕业要求学分
5. 培养方案课程映射禁止重复
6. 推荐学期必须处于 1 到 8

### 12.8 学生培养方案展示流程

1. 根据学生班级找到专业
2. 找到专业绑定的培养方案
3. 查询培养方案中的模块和课程
4. 查询学生所有历史课程进度
5. 计算每门课状态：
   - 通过
   - 未通过
   - 已选课
   - 未修读
6. 聚合：
   - 总学分
   - 已修学分
   - 在修学分
   - 未通过课程数
   - 完成百分比
7. 返回学生端地图页面

## 13. 关键业务规则

### 13.1 选课规则

- 只能选当前学期开课
- 开课状态必须是“开放选课”
- 已满课程不可再选
- 不允许时间冲突
- 退课后可重新恢复为已选

### 13.2 成绩规则

- 录入范围必须在 0 到 100
- 平时占比 + 期末占比必须等于 100
- 成绩发布后才计入学生正式成绩和已通过判断
- 占比变化后，全班历史总评同步重算

### 13.3 评价规则

- 一条选课记录最多一条评价
- 课程未结束且成绩未发布时，当前学期课程不能评价
- 历史学期课程可以评价

### 13.4 培养方案规则

- 一个专业最多一个培养方案
- 模块名称在同一培养方案同一学期内唯一
- 同一课程不能重复加入同一培养方案
- 学分会自动同步，不依赖人工手填总数

### 13.5 删除规则

- 默认不使用数据库级联删除
- 删除前先检查依赖数量
- 有依赖时前端必须给出明确提示
- 标准 UI 走限制性删除，避免误删带来三端数据不一致

## 14. 种子数据与演示设计

### 14.1 种子生成方式

当前项目使用 `scripts/seed-db.js` 生成演示数据，而不是把大量 `INSERT` 语句写死在 `seed.sql` 中。

这样做的优点：

- 更容易与 schema 同步
- 更容易生成跨表一致的数据
- 更容易维护复杂的培养方案、开课、成绩和评价关系

### 14.2 当前演示数据特征

种子数据已经包含：

- 多个学期
- 多个院系与专业
- 多个班级
- 多名教师与学生
- 当前学期与历史学期开课
- 已发布成绩与待录入成绩
- 教学评价
- 学业预警
- 计算机专业培养方案

此外，当前版本还专门补充了：

- 数学与统计学院本学期 5 门真实课程
- 通识教育中心本学期 5 门真实课程
- `t_chen` 当前学期额外承担 3 门计算机学院课程

### 14.3 演示账号

- 管理员：`admin01 / admin123`
- 教师：`t_chen / teacher123`
- 学生：`s_2023001 / student123`

这三个账号可以直接覆盖：

- 管理端维护操作
- 教师成绩录入与发布
- 学生选课、查成绩、看培养方案、做评价
